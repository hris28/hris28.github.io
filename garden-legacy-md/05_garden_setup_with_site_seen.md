# Populating the Garden, with the Site Actually Looked At

I can see your site now. The picture is clearer than I expected and the path forward is also clearer.

## What you have right now

Your site is hand-written static HTML. There is an `index.html`, an `about.html`, a `projects.html`, and so on, with a thin shared header that gives you the navigation strip. This is honest and fast and there is no framework getting in the way of you understanding what is happening on the page. The Garden tab exists as `garden.html` and is currently a one-page placeholder with a description of intent and nothing else. The page promises Obsidian-compatible Markdown, wiki links between entries, and an OSINT-themed information security guide, and right now none of that exists yet because there is no system underneath the page that knows how to read Markdown, render wiki links, or build a graph of connected entries. The placeholder describes a generator's output without a generator behind it.

This is the moment to decide what the garden runs on, and the decision is consequential because you are about to commit to the writing infrastructure that the next two years of your work will sit on top of. Three things are true at the same time. The home page, about page, projects page, and resume page work fine as hand-written HTML and there is no reason to convert them. The garden cannot be hand-written HTML at scale because the wiki links, backlinks, graph view, and pop-over previews all require a build step that reads Markdown, parses link syntax, and produces interconnected pages. The garden therefore needs a generator, and the rest of the site does not. The cleanest way to handle this is to keep what you have for the portfolio pages and run a generator only on the garden directory.

## Why Quartz is the right answer for the garden specifically

The features you said you wanted, in your own words from earlier in this thread, were wiki-link interconnection, hover previews of linked pages, a comprehensive OSINT-themed information security guide, last-reviewed dates surfaced visibly, structured source citations, and a working library of articles that grows over time. These are the features Quartz was built for. Habibullah's SSG comparison post lists Quartz under "Digital Garden" with three native features called out by name: automatic backlinks, graph view, and pop-over previews. The pop-over preview is the feature you asked about. It is not a custom thing you have to build from scratch. It is built in. When a reader hovers over an internal wiki link, Quartz displays the contents of the linked note in a floating panel, without loading the linked page. It does not show a screenshot, which is what you originally said you wanted, but in practice the preview of the actual linked content is more useful than a screenshot of it because the reader can see whether the linked article will answer their question. You will probably end up preferring the content preview to the screenshot version once you use it.

External links are a separate question. Quartz's built-in popover only previews internal pages. For external links to sources, the screenshot-preview approach is still possible, and the right way to do it for the garden's privacy posture is to pre-generate screenshots at build time using a headless browser, store them as static images alongside the post, and render them from your own server. A small tooltip plugin watches for hover events on a custom link class and shows the local screenshot. This is more work than the internal popover, which just works, but it is the same work whether you choose Quartz or any other generator, so it does not change the framework decision.

The other reason Quartz fits is the writing workflow. Quartz reads an Obsidian vault directly. If you are already using Obsidian to take notes, your notes become your garden source files with no conversion step. If you are not using Obsidian, the format Quartz reads is plain Markdown with wiki-link syntax, which any text editor can produce. Either way, you write a note in your normal note-taking flow, copy or symlink the file into the garden's content directory, push to GitHub, and the build action regenerates the static garden. The friction is low. The friction is what determines whether the garden grows.

The other strong contender is Astro, which Habibullah's own portfolio uses and which Tom calls "the modern king" for good reason. Astro is more flexible and produces more visually distinctive sites. The cost is that you would be building the digital-garden behavior yourself: the wiki links, the backlinks, the graph view, the popovers. None of that is impossible in Astro and there are starter templates that do most of it, but you would be choosing complexity that Quartz solves for free. For your situation, where the garden is the artifact and the design polish is secondary to getting the content shape right, Quartz is the better choice. Astro is the right answer if you decide later that you want the garden to look like Habibullah's portfolio rather than like an academic note system. You can switch later if that becomes the priority. The Markdown content survives the migration.

## The actual setup

The minimal version, written so you can do it in an evening, looks like this. Create a new repository called something like `garden` or use a subdirectory in your existing `hris28.github.io` repo. Run `npm create quartz@latest` in that directory. Quartz will scaffold itself, including a `content/` directory where your Markdown lives, a `quartz.config.ts` for the configuration, and a `.github/workflows/` directory with the build action. Replace the default content with your own first article. Push. GitHub Pages serves the result.

The decision to make at this point is whether the garden is at `hris28.github.io/garden/` (a subpath of your existing site) or at `garden.hris28.github.io` (a subdomain) or at its own repo with its own GitHub Pages address. The subpath option is the cleanest from a reader's perspective because the existing navigation strip on your portfolio pages can link to the garden and the garden can link back without anyone needing to remember a separate domain. The subdomain or separate repo are easier to set up because they do not require coordinating two build systems on the same GitHub Pages site. I would recommend the subpath if you are willing to spend an evening on the build coordination, and the separate repo if you want it working tonight. The separate repo can be migrated to a subpath later without losing any URLs if you set it up with this in mind.

For the subpath approach, the trick is that GitHub Pages serves your hand-written HTML from the root of `hris28.github.io` and you want Quartz to serve from `/garden/`. The clean way to do this is to put the Quartz project in a subdirectory of your repo, configure its build action to output to `/garden/` in the published site, and configure the build action on the root site to leave the `/garden/` directory alone. There is friction here and I will not pretend there is not. The separate repo approach is genuinely simpler and produces the same reader experience if your existing `garden.html` is updated to redirect to the new subdomain.

## What the first article looks like

Write your first article as a Markdown file in `content/`. Quartz uses front matter at the top of each file, between two `---` lines. At minimum the front matter should have a title, a date, and a list of tags. For your purposes I would add three more fields right at the start: a description that the index page can use, a last-reviewed date, and a next-review date. The last-reviewed and next-review fields are not standard Quartz fields and you will need to either use Quartz's built-in support for arbitrary front-matter or extend the page template slightly to display them. This is a small amount of TypeScript and well worth doing because it builds the trust posture you described into every page from the start.

The body of the article is just Markdown. Wiki links use the syntax `[[Other Article Name]]` and Quartz handles the linking and the hover preview automatically. External links use standard Markdown syntax. Sources at the bottom of the article use the structured review template we have been working with: URL, publisher, publication date, last-checked date, and a short paragraph on what the source delivered, where it failed, and what its funding is. This is prose, not a table, and it carries through into the rest of the garden as the standard.

The cookies-and-tracking explainer that emerged from your session log is the right first article. Most of the content already exists in your raw data and the topic earns its place in the garden. The visibility table is the right second piece of content because it is the kind of reference page that other articles will link to repeatedly. The third piece should be a tool review, structured similarly to how the papersflow comparison pages are structured but with the source-review discipline added. After three articles in three different shapes, you have validated the templates and the rest is writing.

## What to take from the reference sites

Habibullah's portfolio is doing something interesting, which is committing fully to a visual conceit and letting it carry the whole site. The terminal aesthetic is consistent across every page. The status indicators, the kernel-update-schedule framing of his learning roadmap, the boot-sequence loading screen, the section headers written like shell prompts, all of it commits. The site reads as one person's voice, not as a template. This works because the conceit fits the content. He is a DevOps and security engineer and the terminal aesthetic is what his work looks like.

The conceit would not fit your garden. You are an information science person whose work sits between human-computer interaction, machine learning, and accessible design, and whose research is about how people actually use technology. A terminal aesthetic for a garden about OSINT and information security would be performing technical authority in a way the rest of your work does not. The lesson from his site is not the aesthetic. It is the commitment. Pick a visual language that fits your work, build it consistently across every page, and do not deviate from it. For an academic-leaning OSINT and information-literacy garden, the visual language probably leans toward annotated archive and research notebook. Footnotes that you can click. Source attributions that surface what the source is. Last-reviewed dates that the reader can see without searching. Quartz's default theme is closer to this register than Habibullah's terminal aesthetic, which is another reason it is a fit.

His blog is also worth a closer look. It runs on Hashnode, which is a hosted blogging platform rather than a static site generator. This is the opposite choice from what we are talking about for your garden. The blog gets him search, comments, RSS, and a built-in audience on the Hashnode network for free. The cost is that the content lives on Hashnode's servers under their terms, the URL structure is theirs, and migrating away later means rewriting the URLs. For a garden that is meant to be a long-term archive, the cost is too high. For a side blog where the goal is reach rather than archival permanence, Hashnode is reasonable. You probably do not need both. The garden is the right shape for what you are building.

Habibullah's archive page is the part of his blog worth studying. It lists every post with a date, a brief description, a reading time, and a tag set. The reader scanning the archive can quickly find the post they want and decide whether it is worth their time. Your garden index should do the same things. Quartz's default index lists pages with their dates and titles but not always with descriptions or reading times. Adding a brief description and an estimated reading time to the front matter, and surfacing them on the index, is a small change that significantly improves the reader's ability to navigate. The kits/system page on his portfolio is doing the same thing for tools instead of articles. The format generalizes well.

## The features the reference sites lack

The four features I named in the previous round still apply and Quartz handles the first two cleanly. The visible last-reviewed date is a small front matter and template change. Structured source citations are written discipline rather than a feature, but Quartz's link parser will let you build a small "Sources" component that renders them consistently across articles. The symmetric OSINT framing is editorial rather than technical, which means it is up to you to remember it when you write. The published-research-process artifact is a content choice and Quartz handles it the same as any other Markdown page. None of the four features are blocked by the framework choice.

One feature your reference site list is missing that I would add is a privacy-aware footer. Habibullah's site loads images from Hashnode's CDN, runs analytics, and embeds external scripts. This is normal and fine for a portfolio. For a garden whose entire premise is information literacy and the OSINT throughline, the site itself should walk the talk: no analytics, no third-party fonts loaded from Google, no embedded scripts the reader did not consent to. A small footer line stating "this site does not track you" is verifiable by the reader using the same tools the garden teaches them about. The garden teaching its readers to use Cover Your Tracks and then itself failing the test would be a credibility problem. The garden passing its own test is itself an article you can write and link to.
